# Design: AI Lead Gen Agent — Pain-Signal-First Outreach Platform

Generated by /office-hours on 2026-04-24

Mode: Builder (Hackathon/Demo)

## Problem Statement

AI service companies (AI voice agents, automation tools, chatbot builders) need
outbound sales pipelines but don't want to build them. This platform takes two
inputs — "what AI service do you sell" and "who is your target customer" — and
runs a full lead gen pipeline: discover SMB prospects via web research, detect
a live pain signal for each prospect (bad reviews, broken booking flow, no
after-hours coverage), generate a personalized call script that opens with the
evidence, then place a VAPI voice call. The whole pipeline runs visibly on screen
end-to-end in under 3 minutes.

## What Makes This Cool

"AI found the wound and called with a receipt."

Most outbound tools import a CSV. This one researches in real time, finds a specific
reason to call each prospect TODAY, and leads the call with that proof. Judges watch
it go from two config inputs to a completed call in under 3 minutes — all visible
simultaneously in three pipeline columns.

## Sponsor Tools (4 tools)

- **Guild**: TypeScript-coded agent that orchestrates the pipeline. Investigated
  first (see Next Steps Hour 1). Hosted on Guild's platform, triggered via webhook
  from the frontend. If Guild's external trigger API proves incompatible in Hour 1,
  fallback is to run the agent logic as a plain Next.js API route and still deploy
  to Guild as a hosted agent for judging credit.
- **Insforge**: Backend data layer — Postgres for storing prospect lists, call
  configs, and call logs. Auth + edge functions. Frontend deployed via Insforge
  site deployment. If cut for time, replace with Vercel + Supabase free tier and
  drop to 3 sponsor tools.
- **TinyFish**: Web infrastructure for prospect discovery and pain signal detection.
  Search API for finding prospects. Fetch or Browser API for scraping review/website
  data — which one depends on the Hour 1 GMB prototype (see Open Questions).
  Phone number extraction via structured JSON output.
- **VAPI**: Outbound voice calls triggered by the Guild agent with a structured
  script payload. Call status (initiated / ringing / answered / voicemail / completed)
  streamed back to the frontend via webhook.

## Config Screen (Two Fields)

```
Field 1 — AI Service Description (free text, max 200 chars, required)
  Example: "AI scheduling service that books client consultations for accounting firms"

Field 2 — Target Customer (free text, max 100 chars, required)
  Example: "CPA firms in Memphis TN with 2-15 staff"
```

These two fields drive all TinyFish search queries and the LLM system prompt.

## Agreed Premises

1. Target prospects are SMBs with public phone numbers — Google My Business, Yelp,
   company websites. If a prospect has no extractable phone number, it is skipped
   and the next candidate is used. The pipeline always finds exactly 3 callable
   prospects before proceeding.
2. Core differentiator is pain-signal-first outreach: TinyFish hunts for a live pain
   signal (bad review citing "no answer", missing after-hours link, broken booking
   page) and the script's opening line uses that evidence verbatim. If no pain signal
   is detected for a prospect, fall back to a generic script opening (no prospect
   is dropped solely for missing a pain signal).
3. "Configurable" for the hackathon = the two-field config screen above. Not
   multi-tenant. One service + one ICP per run. The platform story is implied, not
   built.
4. Demo scope: config input → 3 prospects researched + pain signals found →
   3 scripts generated → 1 live VAPI call placed. Total time: under 3 minutes.

## Approaches Considered

### Approach A: Hardwired Demo
One vertical hardcoded. Direct API calls, no orchestration framework. Ships fast,
lowest risk. Weak on sponsor tool coverage and platform story.
Completeness: 6/10

### Approach B: Insforge-Orchestrated (3 tools)
Insforge as backend. Agent logic in custom code. Misses Guild's hosting value.
Completeness: 8/10

### Approach C: TinyFish Agent Live Browse
TinyFish autonomous Web Agent browses live on screen. Maximum wow, maximum risk.
Agent timing unpredictable for a live demo.
Completeness: 7/10

## Recommended Approach: Guild + Insforge + TinyFish + VAPI (4 tools)

**LLM**: Claude (Anthropic API, claude-sonnet-4-6). Called from within the Guild
TypeScript agent. Used for pain signal extraction and script generation.

**Architecture:**

```
[Config Screen]
  service: "AI scheduling service that books client consultations for accounting firms"
  icp:     "CPA firms in Memphis TN with 2-15 staff"
        |
        v
[Guild TypeScript Coded Agent]  ← triggered via webhook from frontend
  1. TinyFish Search: query "[icp]" → top 10 business candidates
  2. For each candidate (parallel, max 3 concurrent):
     a. TinyFish Fetch/Browser: scrape for phone number + pain signals
     b. Filter out candidates with no extractable phone number
     c. Stop when 3 callable prospects are confirmed. If initial 10 candidates
        yield fewer than 3 with phone numbers, proceed with however many are
        available (minimum 1 required to place a call; demo continues).
  3. For each of 3 prospects:
     a. Claude: extract top pain signal → produce script JSON
     b. Script JSON: { opener: string, pain_hook: string, service_pitch: string,
                       objection_answer: string, cta: string }
        (pain_hook is empty string if no signal found — not a failure)
     c. Save prospect + script to Insforge via edge function
  4. Trigger VAPI outbound call for prospect #1
     Payload: VAPI firstMessage = pain_hook ? `${opener}. ${pain_hook}` : opener
     (plain text, per VAPI docs — conditional avoids trailing ". " when no pain signal)
     Stream call status: VAPI webhook → Next.js POST /api/vapi-status →
     in-memory Map (keyed by run ID) → Next.js GET /api/status SSE stream → browser
     (Edge functions are stateless — cannot hold open SSE connections. SSE lives
     in a persistent Next.js server deployed on Vercel/Railway/Render.)
        |
        v
[Three-Column Frontend]  ← deployed via Insforge
  [Researching]        [Writing Script]       [Calling]
  Memphis CPA Group...     Script ready...        Call: initiated → ringing → answered
  Shelby Tax Advisors...   Script ready...        Waiting
  River City Accounting... Generating...          Waiting

Call status states shown in UI:
  initiated → ringing → answered → completed
  voicemail → UI shows "Left voicemail — campaign continues"
  no-answer → UI shows "No answer — will retry"
  (Demo does not fail on voicemail; voicemail is shown as a valid outcome)
```

**Run ID data flow:**

```
1. Frontend generates a UUID (run_id) at form submit
2. run_id is passed in the webhook body to Guild: { run_id, service, icp }
3. Guild stores run_id → used as the key for all status updates
4. VAPI call initiated with run_id in metadata
5. VAPI webhook → Next.js POST /api/vapi-status { run_id, status }
6. In-memory Map[run_id] = status → GET /api/status?run_id=X SSE stream → browser
7. Frontend subscribes to /api/status?run_id=X immediately after form submit
```

This gives the browser a stable key before any async work starts.

**TinyFish fetch timeout:** Each Fetch/Browser call must have a 10-second hard timeout.
On timeout: mark prospect as skipped, log warning, continue with remaining candidates.
If all 10 candidates timeout: abort pipeline, surface "Discovery failed — try again" in UI.

**Claude JSON validation and fallback:**

```typescript
// Per prospect, max 1 retry:
try {
  const raw = await claude(systemPrompt, userPrompt);
  const script = JSON.parse(raw);
  validateScriptSchema(script); // throws if missing keys
  return script;
} catch {
  // retry once with stricter prompt ("respond ONLY with valid JSON")
  try {
    const raw2 = await claude(strictPrompt, userPrompt);
    return JSON.parse(raw2);
  } catch {
    // fallback: generic template script (no pain_hook, stock CTA)
    return { opener: genericOpener(prospect), pain_hook: "", ... };
  }
}
```

The fallback template means the pipeline never stalls on Claude; it just degrades gracefully
(generic script instead of personalized one — still makes the call).

**Fallback if Guild external trigger is too complex (discovered in Hour 1):**
Run pipeline logic in a Next.js API route. Deploy the same TypeScript to Guild
as a manually-triggered agent (judges see it in Guild UI = sponsor credit).
The agent does not need to be the trigger point to count as a sponsor tool.

**What to cut if time runs short (ranked by impact vs. effort):**
1. Drop Insforge → go to 3 tools, use in-memory state + Vercel deploy.
      SSE is always Next.js (edge functions can't hold SSE connections).
      VAPI webhook → Next.js POST /api/vapi-status → in-memory Map → GET /api/status SSE.
2. Drop pain signal detection → generic research-only scripts (still impressive)
3. Drop prospect #3 → research 2 prospects, call 1 (reduce TinyFish calls)

## Open Questions (Resolve in Hour 1 — Blocking)

1. **Guild trigger API**: Can a Guild TypeScript coded agent be triggered via HTTP
   webhook from an external frontend? Check docs.guild.ai for outbound trigger docs.
   If not: use Guild as a hosted runner, trigger from Next.js API route.

2. **TinyFish on Google My Business**: Does TinyFish Fetch handle GMB pages
   (JavaScript-rendered, Google bot-blocking)? Prototype this in Hour 1 before
   any other integration work. If Fetch fails: use TinyFish Browser API (stealth
   Chrome). If Browser also fails: pivot to Yelp-only for reviews, GMB for phone
   numbers only via Search result snippets.

3. **TinyFish fetch latency**: Time a single Fetch call against a real business
   page in Hour 1. Pre-cache prospects 2 and 3 by default — only live-research
   prospect 1 during the demo. Do not wait until demo day to discover latency.
   The "live research" story holds for prospect 1; others show as "already queued."

4. **VAPI outbound number provisioning**: Before Hour 6, confirm a VAPI outbound
   phone number is provisioned in the VAPI workspace and can dial outbound. This
   is a prerequisite — if it's not done before Hour 6, Hour 6 is blocked.
   Do this during account setup on Day 1.

5. **VAPI firstMessage format**: Confirmed as plain text string per VAPI outbound
   call docs. Concatenation rule: `pain_hook ? \`${opener}. ${pain_hook}\` : opener`
   (period + space only when pain_hook is non-empty — avoids trailing ". " vocalized
   by VAPI when no signal is found). Bake this conditional into the Guild agent
   before writing the LLM prompt.

## Performance: Latency Budget

Total target: < 3 minutes from config submit to call completed/voicemail.

```
Step                          Target    Notes
──────────────────────────────────────────────────────────────────────────
TinyFish Search (1 call)       2s       Returns top 10 candidates
TinyFish Fetch × 3 (parallel)  10s      10s hard timeout per call
Claude × 3 (parallel)          6s       ~2s per call; run after fetches resolve
Insforge save × 3              1s       3 concurrent upserts
VAPI call initiation           1s       REST call to VAPI; async from here
VAPI ring / answer / voicemail 60–90s   Actual call duration
UI render overhead             ~0s      SSE push; no polling delay
──────────────────────────────────────────────────────────────────────────
Total (optimistic)             ~80s     ~1:20 — well within 3 minutes
Total (pessimistic w/ retries) ~150s    ~2:30 — still within 3 minutes
```

**Risk: TinyFish latency.** If real-world Fetch against Yelp/GMB pages hits 15-20s
per call (seen in some headless browser scenarios), 3 parallel calls = 20s, not 10s.
Mitigation (from Open Question 3): pre-cache prospects 2 & 3 before demo.
Only prospect 1 is live-researched during the demo presentation.

**Risk: VAPI call duration.** A real call that is answered and runs to 2-minute
conversation could push total to 3:30+. Mitigation: keep VAPI script tight (< 60s
target) and configure VAPI maxDuration = 90 seconds.

**No N+1 patterns.** TinyFish fetches and Claude calls are already parallelized.
Insforge saves are upserts by run_id + prospect index — no sequential chain.

## Test Plan (Demo Mode)

Three tests that must pass before demo day. No TDD, no coverage targets — just demo reliability.

**T1 — UI smoke (Hour 2, hardcoded data)**
Load the three-column frontend with static fixture data. Verify all three call status
transitions render: initiated → ringing → answered, voicemail, no-answer.
This is gating: no integration work starts until this passes.

**T2 — TinyFish prototype (Hour 1, blocking)**
```
curl TinyFish Fetch against: https://www.yelp.com/biz/[real-memphis-cpa-firm]
Assert response contains: phone_number (non-null), at least 1 review text
Assert wall-clock time < 10 seconds
```
If this fails, switch to TinyFish Browser API. If that also fails, fall back to
Yelp search snippet extraction (no full page fetch). Know before Hour 2.

**T3 — firstMessage conditional**
```typescript
// opener does not end in period — the ". " is added by concatenation
const f = (opener: string, hook: string) =>
  hook ? `${opener}. ${hook}` : opener;

assert f("Hi Jane", "I saw you miss calls after 5pm") ===
  "Hi Jane. I saw you miss calls after 5pm";
assert f("Hi Jane", "") === "Hi Jane";  // must NOT be "Hi Jane. "
```
Simple node script, run before wiring VAPI. Catches the trailing-period bug at
zero cost.

**T4 — End-to-end timing (Final hours)**
Run the full pipeline 3 times with the locked niche (CPA firms, Memphis).
Pass criteria: pipeline completes in < 3 minutes. All 3 runs. No exceptions.
If any run exceeds 3 minutes: pre-cache prospects 2 & 3 (see Open Question 3).

## Success Criteria

- Pipeline runs end-to-end in under 3 minutes from config submit to call completion
- At least 1 live VAPI call completes or reaches voicemail during the demo
- Pain signal is specific and quoted in the call script for at least 2 of 3 prospects
- Pipeline is visible on screen at all times — no blank/loading states longer than 5s
- All 4 sponsor tools are visibly used and nameable to judges
- Demo does not visibly fail on voicemail, no-answer, or missing pain signal

## Distribution Plan

- Frontend: deployed via Insforge site deployment
- Agent: hosted on Guild's platform (app.guild.ai workspace)
- Fallback hosting: Vercel (if Insforge deployment causes issues)
- No CI/CD for hackathon — manual deploy is fine

## Not in Scope (Hackathon)

- Multi-tenancy or user accounts (one service + one ICP per run)
- Authentication on the pipeline webhook (acceptable for demo)
- VAPI webhook signature verification
- Rate limiting or abuse prevention
- Automated CI/CD (manual deploy is fine)
- Mobile responsiveness
- Prospect deduplication across runs
- CRM integration beyond Insforge Postgres storage

## Failure Modes and Mitigation

| Failure | Mitigation |
|---------|-----------|
| Guild webhook trigger not supported | Fallback: Next.js API route; deploy same code to Guild for credit |
| TinyFish returns 0 results for query | Broaden ICP query ("accounting firms Memphis" → "CPA Memphis") |
| All 10 candidates have no phone number | Surface error in UI — extremely unlikely for US SMBs |
| TinyFish Fetch fails on GMB/Yelp | Switch to TinyFish Browser API (stealth Chrome) |
| Claude returns invalid JSON twice | Use template fallback script (generic opener, empty pain_hook) |
| VAPI number not provisioned | Hard block on Hour 6 — must do on Day 1 |
| VAPI call goes to voicemail | UI shows "Left voicemail — campaign continues" (valid demo outcome) |
| VAPI call no-answer | UI shows "No answer — will retry" (valid demo outcome) |
| Insforge unavailable | Cut to Fallback #1: in-memory state + Vercel deploy |
| Total pipeline > 3 minutes | Pre-cache prospects 2 & 3; only live-research prospect 1 during demo |
